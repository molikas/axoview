## Goal

Close the two HIGH-severity test gaps named in §11:
- **5a — Server-runtime zero-jest gap.** Stand up jest config for `axoview-backend` + `axoview-worker`; produce contract tests against the framework-agnostic handlers in `routes.js` + the storage contract in `fs.js`; produce middleware tests for `auth.ts` (`isPublicRoute` + mode-switch + JWT verify happy/sad paths).
- **5e — Canvas cross-interaction E2E.** Extend `axoview-e2e` with Playwright specs that exercise the subsystem boundaries where PR #6's regressions lived (lasso × connector × delete-cascade; placeIcon × isoMath; multi-select × drag × collision; mode transitions). The seven scope-rows from the deleted catalogue are the working list.

**Why now:** the audit named both as HIGH severity. PR #6 (2026-05-25) demonstrated empirically that the 5e gap ships regressions; the 5a gap is the structural inversion (most-load-bearing surface, zero tests). Closing them before v1.1 feature work starts means subsequent features land on a tested floor.

**Explicit non-goals:**
- Tier 2 (axoview-app surfaces) and Tier 3 (lib large surfaces + knip B-10 promotion) — catalogued in "Future tiers"; not scaffolded.
- Refactors of any source file touched while writing tests (Track 6 territory; if a test surfaces an unreachable branch or contract violation, it's a finding, not a refactor in this tactical).
- New ADRs. If a contract test surfaces an ADR-worthy bug (e.g., adapter contract violation, error-UX divergence between runtimes), file as a finding; the ADR is a separate vehicle.
- B-9b silent-failure retrofit (S2–S20) — orthogonal; opportunistic per-PR.

## Decision points — locked 2026-05-25

All five orchestrator recommendations accepted by user. Original options retained below for traceability; the `User answer` line under each names the locked choice.

### Decision Point 1 — Which Tier 1 subtask first: 5a or 5e?

- **(a) 5a first** *(recommended)* — server-runtime contract tests; Node-side, deterministic, ~620 LOC of source surface, scoped to 4 files. Locks harness conventions cleanly before 5e's larger surface area lands.
- **(b) 5e first** — closes the PR #6 class-of-bug gap immediately; Playwright surface is closer to user-visible; larger surface area; lib-rebuild churn per anchor.
- **(c) Parallel** — rejected; both PRs introduce significant new test scaffolding and reviewing them concurrently is friction.

**User answer:** **(a) 5a first** — locked 2026-05-25.

### Decision Point 2 — Contract-test framework for 5a's backend + worker

Neither backend nor worker has a jest config today. The repo's existing test runner is jest (lib + app). The Worker is TypeScript (ts-jest) and the backend is plain JS.

- **(a) jest + supertest (backend) + Hono `app.request` (worker)** *(recommended)* — single runner across the repo, two adapter shells inside the harness. Convention match with lib + app; Hono's `app.request` is in-process and zero-dep.
- **(b) vitest + MSW for both** — modernizes runner; introduces a second runner; loses parity with lib + app jest configs.
- **(c) jest with a shared minimal HTTP harness** — one suite drives both adapters via thin wrappers; premature given the §12 B2 reality that the two runtimes don't share routes at runtime.

**User answer:** **(a) jest + supertest (backend) + Hono `app.request` (worker)** — locked 2026-05-25.

### Decision Point 3 — Playwright selector strategy for 5e

The canvas itself has no per-tile DOM (it's an SVG/canvas with tile coords computed by isoMath). Existing canvas specs mix `data-axoview-id` (canonical per ADR 0008 D5), `byLibTestId` (transitional), and coord-based mouse dispatch via CanvasPOM.

- **(a) Lazy `data-axoview-id` retrofits + coord-based mouse for canvas** *(recommended)* — current convention; minimal new attributes; each new id is a lib rebuild. Cross-interaction tests are mostly canvas-coord-driven anyway (lasso, drag, click); likely 2–3 new ids across all 6 specs.
- **(b) Mixed — `data-axoview-id` + role-based queries** — augments with `getByRole` where i18n labels are stable; introduces i18n-stability assumptions that aren't worth it for 5e's scope.
- **(c) Coord-based mouse dispatch only via CanvasPOM** — no new DOM attributes; selectors live in POM helpers as tile-coord → screen-coord conversions.

**User answer:** **(a) Lazy `data-axoview-id` retrofits + coord-based mouse for canvas** — locked 2026-05-25.

### Decision Point 4 — Fixture management for 5e

Current pattern is per-spec inline `bootBlankDiagram` + onboarding-dismiss flags (duplicated across `canvas-modes.spec.ts`, `connector.spec.ts`, others). Track 0d deleted an unused `canvas.fixture.ts`.

- **(a) Per-test factories (status quo)** — copy the `bootBlankDiagram` + onboarding flags pattern into each new 5e spec.
- **(b) Shared `app.fixture.ts` extension** *(recommended)* — add a `canvasReadyTest` extension that does boot + onboarding-dismiss once; new specs consume it. Six immediate consumers justify centralizing.
- **(c) Per-scope fixtures** (`canvas-with-2-items.fixture.ts`, etc.) — over-fitted given the unique scenarios per spec.

**User answer:** **(b) Shared `app.fixture.ts` extension (`canvasReadyTest`)** — locked 2026-05-25.

### Decision Point 5 — Shared contract suite vs. parallel suites for 5a (may merge into DP2)

§12 B2 corrected the audit's "single source of truth" claim — `app.ts` does NOT import `routes.js` at runtime. The "shared contract" between Express and Worker exists only at the HTTP-shape level by convention.

- **(a) Parallel suites** *(recommended)* — `routes.js` tested directly with an in-memory adapter; `app.ts` tested as its hardcoded 503 + config-only surface. No "shared interface" abstraction. Honours the §12 B2 reality.
- **(b) Shared contract suite** — one parameterized suite runs the HTTP contract twice (Express adapter, Hono adapter), with Worker 503 noted as a known-skip per storage route. Builds an abstraction that pretends both runtimes serve the storage contract when only Express does — exactly the inversion §12 B2 flagged.

**User answer:** **(a) Parallel suites** — locked 2026-05-25.

## Per-track detail

### 5a — Server-runtime contract tests

**Surface inventory:**
- `packages/axoview-backend/src/routes.js` — 325 LOC; exports `HttpError` + handler factories. Every handler takes `(adapter, ctx)` and returns `{status, body}` or throws `HttpError`. Validation helpers `assertId` / `assertUuid` + UUID/ID regexes inline.
- `packages/axoview-backend/src/adapters/fs.js` — 95 LOC; `createFsAdapter(storagePath)` returns the five-method `StorageAdapter` shape per ADR 0010. Key→path mapping via `KEY_PATTERN` (defense-in-depth). Atomicity via tmp+rename per ADR 0010 D3.
- `packages/axoview-worker/src/app.ts` — ~25 LOC; Hono entry. `GET /api/config` inlined (does NOT call `routes.js`); `app.all('/api/*', ...)` returns 503.
- `packages/axoview-worker/src/auth.ts` — ~175 LOC; `isPublicRoute` + `authMiddleware` + JWKS-cached RS256 verifier.

**Test plan (subject to DP2 + DP5):**
- `routes.js`: per-handler suites with an in-memory adapter (`Map<string, Uint8Array>` honouring the five-method shape). Cover happy path + every documented `HttpError` (`400` invalid id, `404` missing, etc.). Use the actual `getConfig` shape (assert `serverStorage: env.STORAGE_ENABLED !== false`).
- `fs.js`: temp-dir per test (jest `tmpdir`). Cover get/put round-trip; missing-key returns null; `KEY_PATTERN` rejection; atomicity (kill mid-write is hard to simulate cross-platform — assert the tmp-file artifact is cleaned up post-success); concurrent `put` races (sequential here; race semantics are a finding if surfaced).
- `auth.ts`: table-driven `isPublicRoute` cases; mode-switch (`none` / `shared-token` valid / `shared-token` mismatch / `shared-token` misconfigured / `cf-access` misconfigured); JWT verify path with a stubbed JWKS + stubbed `crypto.subtle.verify` returning true/false.
- `app.ts`: Hono `app.request()` for `GET /api/config` (assert documented shape); `app.request()` for a representative storage route (assert 503); auth-middleware-applied (assert 401 for non-public route with no auth in `shared-token` mode).

### 5e — Canvas cross-interaction E2E

**Surface inventory:** `packages/axoview-e2e/tests/` (~13 specs, ~33 tests today). Existing canvas-touching specs: `canvas-modes.spec.ts` (J19 — projection toggle), `connector.spec.ts` (J2 — single connector draw + undo), `shapes.spec.ts` (J3 — rectangle + textbox), `layers.spec.ts` (J6 — layer drag), `hotkeys.spec.ts` (J15 — hotkey table). None of these exercise cross-interaction (lasso × connector × delete; placeIcon × isoMath after a partial-delete; mode-transition state-leak; multi-select + drag with collision).

**Test plan (subject to DP3 + DP4):**
- `lasso-connector-delete.spec.ts` — the PR #6 scenarios (regression test layer). Partial connector-anchor lasso → delete → place node (exact user-reported repro PR #6 fixes; assert it doesn't throw at the E2E layer). Path-hit selection (lasso crosses a connector whose endpoints are outside).
- `connector-creation.spec.ts` — between two free tiles, between anchored tiles, mixed; cancel mid-drag (Esc).
- `drag-collision.spec.ts` — node onto occupied tile (rejected vs. swap per current contract — read the lib code to confirm before asserting); onto rectangle footprint; group-drag with collision.
- `z-order.spec.ts` — rectangle pulled under regular nodes; selection on overlap.
- `multi-select-drag.spec.ts` — group move preserves relative positions; undo/redo round-trip.
- `mode-transitions.spec.ts` — lasso → place-icon → connector → pan; assert no stuck mode after each handoff; Esc returns to default cursor mode.

**Anchor strategy:** see DP3 lock. Where new ids are needed, use coord-based mouse dispatch via `CanvasPOM` extensions; new POM methods land lazily per `pom/_pending.md`. Each new `data-axoview-id` retrofit is a small lib edit + rebuild; budget accordingly.

## Notes for Claude

- **§12 B2 correction is load-bearing.** Don't write contract tests on the assumption that `app.ts` imports `routes.js`. It doesn't. Verify with `grep` before any cross-runtime-shared-contract test design.
- **Dev-server lib-rebuild friction applies to 5e.** Any new `data-axoview-id` retrofit in `axoview-lib` requires `npm run build:lib` + restart of `npm run dev` (memory: dev-server reads lib from `dist/`, not source). Budget the restart into every 5e session.
- **PowerShell encoding hazard** if any test fixture is non-ASCII. Use `Out-File -Encoding utf8NoBOM` or write from bash. Never `Set-Content -Encoding utf8` on Unicode-bearing files.
- **PR #6's commit message is the canonical scope statement for 5e.** Don't paraphrase from this tactical — read the original.
- **`pom/_pending.md`** is the audit trail for lazy `data-axoview-id` retrofits. Update it when a new one lands; don't sweep ahead.
- **Worker is hardcoded 503.** 5a assertions on `app.ts` must mirror this. If a future ADR un-503's the worker, that's a separate tactical's problem.
- **No new ADRs from this tactical.** If a contract test surfaces something ADR-worthy, file as a finding row + escalate to user; the ADR vehicle is `/feature start`, not inline edits here.
- **Frozen artifact convention.** `technical-review-2026-05.md` §1–§11 is frozen; only §12 receives append-only corrections.
- **One PR per Tier 1 subtask.** 5a = one PR (~5–7 commits per the first-session prompt). 5e = one PR. Don't bundle.
