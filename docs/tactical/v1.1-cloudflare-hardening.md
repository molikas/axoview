## Architecture context (re-framing the CF Analytics review)

The original CF dashboard review classified findings against a "traditional origin server" mental model. Axoview prod is Cloudflare Pages + Functions, which reframes both real concerns:

- **`/api/*` paths** hit the Worker via `functions/api/[[path]].ts`. Per ADR 0009 D1 + §12 B2, `app.ts` is hardcoded 503 for everything except `GET /api/config` (inlined). A 500 here means `auth.ts` middleware throws on a specific input OR the catch-all itself throws OR a runtime exception escapes the Hono router. **Diagnosis vector:** extend `app.spec.ts` with the actual probe-path inputs the CF dashboard recorded.
- **Non-`/api/*` paths** (`/.env`, `/wp-config.*`, `/.docker/secrets.json`) hit Pages static serving directly (the Worker is scoped out by `_routes.json`). A 500 likely means SPA-fallback routes the request to `index.html`, and something in the SPA boot throws on these inputs. **Diagnosis vector:** verify what static serving actually returns for these paths today (curl the deployed origin).
- **522 timeouts** on CF Pages + Functions are a Worker cold-start exceeding the edge timeout window. Different diagnosis class than "traditional origin intermittent unreachability." **Diagnosis vector:** check `wrangler tail` for the 522-emitting paths' invocation times; cross-reference Worker bundle size against ADR 0009 D8's < 1 MB budget.

The original review's P1–P4 priorities stand, but their *implementations* differ:

| Original P# | Original framing | This-tactical framing |
|---|---|---|
| P1 | "Investigate 5xx root cause origin-side" | Workstream A — extend `app.spec.ts` with probe inputs; reproduce locally with `wrangler dev`; identify whether the 500 originates in auth, the 503 catch-all, or SPA boot. |
| P2 | "Diagnose 522 timeouts" | Workstream A.2 — `wrangler tail` sampling + bundle-size check; likely a cold-start finding, not an availability finding. |
| P3 | "CF edge hardening (BFM / WAF / scanner block / rate limit)" | Workstream B — agent prepares WAF rule expressions + rate-limit config + dashboard click-path docs; user executes in CF UI; agent verifies via post-deploy curls. |
| P4 | "Verify no actual exposure of probe paths" | Hygiene — short checklist; deferred behind A + B. |

## Pending Decision Points (block first execution session)

These need user answers before Workstream A's first prompt is fully scoped. They are surfaced in the orchestrator's return message and reproduced here for traceability:

| DP# | Question | Recommendation |
|---|---|---|
| DP1 | **WAF block rule tightness.** Exact-match probe paths only (a static enumeration the agent will produce from the CF analytics paths list — `/.env`, `/api/.env`, `/api/v2/.env`, `/.docker/secrets.json`, `/api/config.js`, `/api/node/constant.js`, `/api/admin/role/id`, `/api/v1/executions`, `/wp-config.php~`, `/wp-config.php.old`, `/.git/config`, `/graphql`, `/graphql/api`, `/__nextjs_action`, `/_next/*`) vs. broader `ends_with(".env")` + `ends_with(".old")` + `ends_with("~")` regex families vs. a combination? | **Combination:** path-exact-match list for the known fingerprints (zero risk of false positive) + a narrow `ends_with(".env")` + `ends_with(".old")` + `ends_with("~")` family (catches future variants of the same scanner). Decision needed before Workstream B drafts the rule expressions. |
| DP2 | **Rate-limit threshold for unauthenticated routes.** The original review suggested `100 req/min` blanket on unauthenticated paths. Should we accept that or calibrate against the legitimate-traffic floor? | **Accept 100 req/min** as the v1 number — Axoview is single-tenant per deploy (ADR 0010 D4); a legitimate user does not exceed 100 req/min on the public-share endpoint or `/api/config`. Calibration is a follow-up if a legitimate user reports being rate-limited. |
| DP3 | **Honey-pot 404 vs. clean SPA fallback for non-`/api/*` probes.** The current SPA fallback (presumably) renders `index.html` for unknown paths, then the SPA's router shows the diagram-not-found state. Should we deploy a deliberate 404 handler that "stops answering" probes (might confuse a legitimate user who typos a URL) or keep clean SPA fallback (continues to serve the SPA boot to probes, which is the current shape)? | **Keep clean SPA fallback for v1; revisit if Workstream A confirms the SPA boot is the 500 source.** A honey-pot adds a new code path with its own bug surface; the WAF rules in Workstream B will short-circuit known probes at the edge anyway. |
| DP4 | **Structured logging for 500 responses.** Per ADR 0009 D6, the observability table acknowledges a gap (no `morgan` or structured logger on Express; Worker has no structured logger either). Should Workstream A add `console.log` instrumentation on the Worker's 500 paths so the diagnosis has signal in `wrangler tail`? | **Yes — add a single `console.error` at the Hono `onError` handler with the request path + method + error name (not the stack — stack stays internal).** Lands in the same session as the probe-input tests so the local repro signal matches the production signal shape. |

## Sub-tasks per workstream

### Workstream A — Origin diagnosis (sessions A.1 + A.2)

**A.1 — Worker probe-input coverage + 500 root cause** (first execution session)

- Extend [`app.spec.ts`](../../packages/axoview-worker/src/__tests__/app.spec.ts) with a `describe('probe-input surface (CF analytics 5xx fingerprints)')` block. Inputs:
  - `/.env`, `/api/.env`, `/api/v2/.env` (env-secret probes).
  - `/.docker/secrets.json` (Docker secret probes).
  - `/api/config.js`, `/api/node/constant.js` (embedded-credential probes).
  - `/api/admin/role/id`, `/api/v1/executions` (IDOR / n8n RCE probes).
  - `/.git/config`, `/wp-config.php~`, `/wp-config.php.old` (recon fingerprints — non-`/api/*`, expected 404 by Worker scope, but verify).
  - `/graphql`, `/graphql/api`, `/__nextjs_action`, `/_next/foo` (framework-version probes).
- For each input, assert the expected response status. Hypothesis: `/api/*` probes return `503` (the catch-all); non-`/api/*` probes return `404` (no Worker route).
- If a test reveals a 500 — that IS the diagnosis. Capture the stack in the test failure, attribute to either auth-middleware or catch-all or Hono routing.
- Add a `console.error` at a Hono `onError` handler in [`app.ts`](../../packages/axoview-worker/src/app.ts) (per DP4): log request method + path + error name only. Stack stays internal.
- Reproduce locally with `wrangler dev` (or rely on Jest if the test environment fully covers; document which).
- Self-pause checkpoint: if the test suite passes with all probe paths returning 503/404 cleanly, the 500s observed in production are NOT reproducible from the current Worker code on `integration`. Pause and report — the cause may be a deployed older bundle, an env-mode mismatch, or an SPA-side path (non-`/api/*`) that needs separate investigation.

**A.2 — 522 cold-start cross-reference** (second execution session, may bundle with A.1 if short)

- Sample `wrangler tail` for production for ~5–10 minutes; capture invocation times for any path that 522'd in the analytics window.
- Run a current bundle-size check on the Worker (`npm run build` in `packages/axoview-worker/`, then size the output). Compare against ADR 0009 D8's < 1 MB budget.
- If bundle is well under budget and tail invocation times are well under CF's edge timeout window (currently 100s for Pages Functions on free tier; verify), the 522 hypothesis is NOT cold-start — re-frame to "intermittent CF edge connectivity" and escalate to user.
- If bundle is borderline or tail shows slow invocations on the 522'd paths, identify which import / which middleware step is the long pole. (Likely candidates: `loadJwks` on `cf-access` first request after the 1-hour JWKS TTL expires.)

### Workstream B — CF edge hardening (sessions B.1 + B.2)

**B.1 — Agent prepares rule expressions + click-path docs** (one session)

Agent produces, in this tactical (as an appendix), the following ready-to-paste materials:

1. **Bot Fight Mode** enablement instructions — Cloudflare dashboard path (Security → Bots), screenshot-anchor descriptions for the toggle.
2. **WAF Managed Rules** enablement instructions — Cloudflare Managed Ruleset + OWASP Core Ruleset; dashboard path; ruleset selection guidance.
3. **Scanner-path block rule** — exact CF expression syntax per DP1's answer. Example (subject to DP1 confirmation):
   ```
   (http.request.uri.path in {"/.env" "/api/.env" "/api/v2/.env" "/.docker/secrets.json"
     "/api/config.js" "/api/node/constant.js" "/api/admin/role/id" "/api/v1/executions"
     "/wp-config.php~" "/wp-config.php.old" "/.git/config" "/graphql" "/graphql/api"
     "/__nextjs_action"}) or
   (ends_with(http.request.uri.path, ".env")) or
   (ends_with(http.request.uri.path, ".old")) or
   (ends_with(http.request.uri.path, "~")) or
   (starts_with(http.request.uri.path, "/_next/"))
   ```
   Action: `Block`. Skip this rule for the public-namespace path (`/api/public/diagrams/*`) and `/api/config` — both must remain reachable.
4. **Rate-limit rule** per DP2 — 100 req/min from unauthenticated source IPs on `/api/*` paths excluding `/api/config`; action: `Block` with 429 response.
5. **Dashboard click-path documentation** — step-by-step for the user to execute B.1's output in the CF dashboard.

**B.2 — User executes; agent verifies** (one session)

- Agent runs `curl` against `/api/config` and the public-namespace path; expected: `200` (unaffected by the WAF block).
- Agent runs a burst of 110 requests against `/api/config` from a single IP within 60s; expected: `429` on requests ≥ 101.
- Findings appended to register; PR opens with the verification curl-script committed for future re-runs.

### Workstream A.1 verdict (2026-05-27)

**500 NOT reproducible from current `integration` bundle.** All 15 probe paths return expected statuses (503 for `/api/*` catch-all in `AUTH_MODE=none`; 401 in `AUTH_MODE=shared-token`; 404 for non-`/api/*` via Pages static serving).

**Implication:** the production 5xx signal is NOT in the current Worker code path. Three candidate causes in decreasing likelihood:

1. **Deployed bundle older than current `master`** — most likely. Check via CF Pages dashboard deployment SHA or `wrangler deployments list`.
2. **Non-`/api/*` SPA boot path throwing on probe URLs** — separate code surface; Worker is bypassed for these paths per [`_routes.json`](../../packages/axoview-app/public/_routes.json). Would not be Worker-visible but CF Analytics aggregates Pages + Worker.
3. **`cf-access` env-mode misconfig** — already ruled out: [`auth.ts:68-71`](../../packages/axoview-worker/src/auth.ts#L68-L71) catches verify errors and returns 401, never 500.

**Disposition:** Workstream A.1 ships the onError observability handler + the probe-input contract tests as the durable security improvement. Workstream A.2 (`wrangler tail` + bundle-size cross-reference) **DEFERRED INDEFINITELY** — revisit only if 5xx signal persists in CF Analytics ≥ 3 days post-deploy of current `master`. Workstream B (CF edge configuration) remains deferred per LD #6. Hygiene P4 deferred behind both.

**Recommended user action post-merge:** 30-second deployed-SHA check. If deployed SHA ≠ current `master` → wait for next deploy + observe. If deployed SHA == current `master` AND 5xx noise persists → revisit as a new narrow workstream targeting the SPA boot path (candidate #2 above).
